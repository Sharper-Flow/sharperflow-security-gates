Fully retired SonarCloud across Sharper Flow. Background: the Sonar CI workflows had been removed earlier, but the `sonarqubecloud` GitHub App + `sonar-project.properties` + `SONAR_TOKEN` were still live, so SonarCloud Automatic Analysis kept running. This change completed the retirement: removed `sonar-project.properties` from both app repos (PRs #277 pokeedge, #94 web), deleted the `SONAR_TOKEN` secret from both, reframed pokeedge's CI/quality docs to 'SonarCloud retired', and reconciled the security-gates standard (`ci-standard.md` + `ROADMAP.md`) to state Sonar is retired with the four capability gaps tracked as research followups. The actual analysis-stop ‚Äî uninstalling the `sonarqubecloud` GitHub App (which unbinds the SonarCloud org) ‚Äî is an org-owner UI action not reachable via API (verified: the user PAT gets 404 on the org-installation DELETE); the org owner performed it, and `gh api orgs/Sharper-Flow/installations` confirms `sonarqubecloud` is no longer installed. A key verified subtlety drove the design: removing `sonar-project.properties` does NOT stop analysis (Sonar ignores it under Automatic Analysis) ‚Äî only the App uninstall / Automatic-Analysis disable stops it, so the change deliberately blocked acceptance (AC3/DONT2 left `unknown`) until the App was actually gone. The four Sonar capabilities lost to retirement ‚Äî duplication detection, maintainability/tech-debt metrics, coverage-on-new-code, and deep dataflow/taint SAST ‚Äî are each tracked as a research‚Üídecide‚Üíimplement fast-follow change (`addDuplicationDetection`, `addMaintainabilityMetrics`, `addDiffCoverageGate`, `deepenSastDataflow`); trends/dashboard/PR-decoration are intentionally dropped per the no-hosted-dashboard posture. Contract 10/10 (after the org-owner uninstall flipped AC3 + DONT2 to satisfied).
